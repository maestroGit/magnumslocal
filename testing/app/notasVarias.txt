🧱 Incluir miner en el bloque: ¿sí o no?
✅ Ventajas
- Trazabilidad directa: permite identificar fácilmente quién minó cada bloque.
- Visualización clara: útil para dashboards, exploradores de bloques o interfaces educativas.
- Estadísticas por nodo: facilita análisis de participación, eficiencia y recompensas.
⚠️ Desventajas
- No garantiza veracidad: si no se firma, el campo puede ser falsificado.
- Redundancia: la dirección del minero ya aparece en la transacción de recompensa.
- Escalabilidad: en redes grandes, mantener registros fiables requiere validación adicional.

🧠 ¿Cómo se identifica el minero en redes reales?
- Bitcoin/Ethereum: el minero se infiere por la dirección que recibe la recompensa (coinbase transaction).
- No hay campo miner explícito, pero puede deducirse del primer output del bloque.

🛠 Recomendaciones para tu proyecto
- En redes pequeñas o educativas: incluir miner como campo explícito es útil y práctico.
- Para mayor robustez: validar que miner === rewardTx.output.address.
- En redes más avanzadas: considerar firma digital del bloque por parte del minero.

🧱 Extensión: Visualizador por minero + análisis de outputs
✅ Objetivo
Mostrar los bloques minados por cada dirección de minero verificado, junto con el estado de sus transacciones:
- 🟢 Transacciones con saldo: outputs con valor > 0.
- 🔴 Transacciones vacías: outputs con valor 0 (quemadas, nulas o de control).

🧩 Requisitos técnicos
- Campo miner en el bloque
- Se asigna al crear el bloque: block.miner = address || wallet.publicKey.
- Visualizador por minero
- Agrupa bloques por block.miner.
- Muestra cantidad de bloques, total de transacciones, tiempo promedio.
- Análisis de outputs
- Recorre block.data y filtra transacciones con output.amount > 0.
- Cuenta y muestra las que tienen amount === 0.

🛠 Estructura sugerida
export const renderMinerStats = (chain) => {
  const miners = {};

  chain.forEach((block) => {
    const miner = block.miner || "Desconocido";
    if (!miners[miner]) {
      miners[miner] = { blocks: [], txWithValue: 0, txZero: 0 };
    }

    const txs = block.data || [];
    txs.forEach((tx) => {
      const outputs = tx.outputs || [];
      outputs.forEach((out) => {
        if (out.amount > 0) miners[miner].txWithValue++;
        else miners[miner].txZero++;
      });
    });

    miners[miner].blocks.push(block);
  });

  return miners;
};

🧱 Estructura técnica
- Wallet especial (Coinbase Wallet)
- Se crea mediante Wallet.blockchainwallet().
- Tiene una clave pública fija que actúa como origen legítimo de fondos.
- No depende de claves privadas reales, ya que no firma transacciones en este contexto.
- Wallet destino
- Se instancia con new Wallet() o se define con clave pública específica.
- Recibirá los 1000 como output en una transacción registrada.
- Transacción de inicialización
- Se crea con Transaction.newTransaction(coinbaseWallet, recipientWallet.publicKey, INITIAL_BALANCE).
- No pasa por la mempool ni requiere minería si se añade directamente como bloque.
- Registro en blockchain
- Se añade con blockchain.addBlock([initTx]).
- Esto asegura que el saldo sea trazable y verificable vía calculateBalance() y getUTXOs().
- Condición de ejecución
- Solo se ejecuta si blockchain.chain.length === 1, es decir, si la cadena solo contiene el bloque génesis.
- Evita duplicar saldo en reinicios.

🧪 Validación
- El balance puede consultarse vía:
GET /address-balance
POST /utxo-balance/:address
- El bloque de inicialización aparece en /blocks con la transacción correspondiente.

🧠 Ventajas
- ✅ Trazabilidad total: el saldo está respaldado por una transacción registrada.
- ✅ Compatible con el modelo UTXO: el balance se calcula desde los outputs.
- ✅ No rompe el flujo de validación ni la lógica de minería.
- ✅ Permite extender el sistema para inicializar múltiples wallets desde JSON o scripts.
